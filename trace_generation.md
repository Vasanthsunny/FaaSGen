# Trace Generation and Workflow Execution Monitoring

After all workflows are successfully deployed and verified, execution traces can be generated for benchmarking, scheduling evaluation, and performance analysis.

The repository includes a dedicated trace/ directory for automated workload generation and trace extraction.

---

## Trace Directory Structure
```text
trace/
├── config.yaml
├── generate_trace.py
├── extract_trace.py
├── dump.sh
├── sample_trace.csv
└── data/
```
---

## Trace Components

config.yaml

Main configuration file used for workload generation and workflow execution control.

This file defines:

- Workflow endpoints
- Request generation parameters
- Arrival rates
- Burst traffic settings
- Cold-start behavior
- Payload types
- Input data directories

## Example configuration:

global:
  gateway_url: "http://<HOST_IP>:31519"
  total_requests: 500
  arrival_rate: 1
  timeout: 40
  max_in_flight: 50
  warmup_requests: 10
cold_start:
  enabled: true
  idle_threshold: 30
burst:
  enabled: true
  multiplier: 3
  interval: 60
  duration: 10
workflows:
  - name: tg_workflow
    host: tg-f1-gen.default.<HOST_IP>.nip.io
    content_type: application/json
    payload_dir: ./data/tg_workflow/tg_requests/
    payload_mode: json_file

---

## Data Directory

The data/ directory contains all workflow-specific request payloads.

Example:

```text
data/
├── tg_workflow/
├── qr_workflow/
├── sec_workflow/
├── img_workflow/
├── etl_workflow/
├── ocr_workflow/
├── fd_workflow/
├── ecom_workflow/
├── iot_workflow/
├── mc_workflow/
└── log_workflow/
```
Each workflow directory contains JSON request payload files used during execution. for MLT workflow, compresssed (ZIP) file should be supplied (images in disticnt directories for each type).

Example:
```text
{
  "request_id": "101",
  "input": "sample text"
}
```

# Generating Workflow Traces

## Step 1 — Deploy All Workflow Pods

Ensure all workflows are successfully deployed and accessible through Knative.

Example:

```bash 
kapply_all.sh
```
Verify services:

```bash 
kubectl get ksvc
```
---

## Step 2 — Configure Workflow Endpoints

Before generating traces:

- Replace all placeholder IP addresses with the actual host/master node IP address.

Example:

192.168.1.100

This replacement must be applied in:

- trace/config.yaml
- Knative service URLs
- Any hardcoded workflow endpoint references

---

## Step 3 — Generate Requests and Execute Workflows

Run:

```bash 
python3 generate_trace.py
```
This script:

- Sends requests to workflow entry functions
- Generates workload traces
- Simulates concurrent executions
- Supports cold-start benchmarking
- Supports burst workload generation
- Uses workflow-specific JSON payloads

---

## Trace Dump Extraction

dump.sh

After workflow execution, Redis stores execution metadata generated by workflow functions.

The script:

```bash
dump.sh
```

extracts Redis dump files containing runtime execution details.

---

## Important Configuration Changes

Before running dump.sh:

Update Redis Pod Name

Modify the Redis pod name inside:

dump.sh

Example:

REDIS_POD=redis-master-0

Replace it with the actual Redis pod running in the Kubernetes cluster.


---

## Trace Record Format

Each workflow function generates records similar to:

row = {
    "function": safe_str(data.get("function")),
    "arrival_time": safe_float(data.get("arrival_time")),
    "start_time": safe_float(data.get("start_time")),
    "end_time": safe_float(data.get("end_time")),
    "processing_time_ms": safe_float(data.get("processing_time_ms")),
    "cpu_time_sec": safe_float(data.get("cpu_time_sec")),
    "memory_mb": safe_float(data.get("memory_mb")),
}

These records are stored in Redis during execution.

---

## Extracting CSV Traces

Run:

```bash
python3 extract_trace.py
```

This converts Redis dump files into standard CSV traces for analysis.

Generated traces contain:

Field	Description
function	Function name
arrival_time	Request arrival timestamp
start_time	Function execution start time
end_time	Function execution completion time
processing_time_ms	Total processing latency
cpu_time_sec	CPU execution time
memory_mb	Memory usage during execution

## Experimental Notes

- Cold-start experiments require scale-to-zero enabled in Knative.
- Burst traffic can significantly increase queueing delays.
- MinIO object references reduce payload transfer overhead for large files.
- Some Workflow DAGs contain parallel execution branches.
- Redis is used for lightweight trace aggregation and temporary runtime storage.
